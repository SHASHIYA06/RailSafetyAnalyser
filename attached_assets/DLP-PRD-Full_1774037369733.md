## 3. PROBLEM STATEMENT & CURRENT CHALLENGES

### Challenge 1: Manual Inventory Tracking

**Issue:** All inventory tracked in Excel spreadsheets updated manually  
**Impact:** 
- High error rate (manual data entry mistakes)
- Delays in identifying shortages (days instead of minutes)
- Duplicate entries causing confusion
- Lost visibility when key people on leave

**Business Impact:**
- Unexpected parts shortages causing train maintenance delays
- Emergency procurement at 40-50% premium costs
- Compliance audit findings for documentation gaps

### Challenge 2: No System Integration

**Issue:** DLP inventory system is completely siloed from NCR, Job Card, and Gate Post systems  
**Impact:**
- Manual cross-reference required for every parts request
- No automatic parts allocation when NCR created
- Parts usage not linked to maintenance tasks
- No traceability of item movements

**Business Impact:**
- Maintenance team unable to complete work due to unavailable parts
- Hidden dependency chains not identified
- Compliance violations for missing documentation

### Challenge 3: Vendor & Cost Management

**Issue:** No centralized vendor master or cost tracking  
**Impact:**
- Duplicate vendor records in different systems
- No consolidated delivery performance tracking
- Cost tracking manual and delayed
- No data for strategic vendor negotiations

**Business Impact:**
- Budget overruns due to untracked procurement
- Missed negotiation opportunities for volume discounts
- Sub-optimal vendor performance not identified

### Challenge 4: No Predictive Analytics

**Issue:** No forecasting or trend analysis on parts consumption  
**Impact:**
- Frequent emergency procurements
- Stock-outs of critical items
- Over-stocking of slow-moving items
- Working capital tied up unnecessarily

**Business Impact:**
- Maintenance schedule disruptions
- Budget variance of 15-20%
- High carrying costs for excess inventory

### Challenge 5: Audit & Compliance Gaps

**Issue:** No complete audit trail or compliance documentation  
**Impact:**
- Difficult to trace parts history
- Accountability gaps in custody
- Compliance audits require weeks of manual investigation
- Regulatory findings on documentation

**Business Impact:**
- Potential regulatory penalties
- Loss of audit confidence
- Difficulty in RFQ and tender processes

## 4. SOLUTION OVERVIEW

### Core Concept

A centralized, cloud-connected DLP Store Inventory Management System that:

1. **Automates** parts tracking and management workflows
2. **Integrates** seamlessly with existing NCR, Job Card, and Gate Post systems
3. **Provides** real-time visibility and analytics
4. **Enables** complete audit trail and compliance

### Architecture Layers

```
┌─────────────────────────────────────────────────────────────────┐
│                    User Interface Layer                         │
│  (Web App: React, Mobile: Progressive Web App)                  │
└──────────────────────────┬──────────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────────┐
│                  Application Layer                              │
│  (APIs, Business Logic, Workflows, Integration Handlers)        │
└──────────────────────────┬──────────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────────┐
│                    Data Layer                                   │
│  (PostgreSQL, Redis Cache, Elasticsearch)                       │
└──────────────────────────┬──────────────────────────────────────┘
                           │
┌──────────────────────────▼──────────────────────────────────────┐
│            External System Integration                          │
│  (NCR, Job Card, Gate Post, Financial Systems)                  │
└─────────────────────────────────────────────────────────────────┘
```

### System Boundaries

**In Scope:**
- Complete DLP item master management
- Real-time stock level tracking
- Transaction management (issue, receipt, return, transfer)
- NCR, Job Card, and Gate Post integration
- Search, filter, and export capabilities
- Standard and custom reporting
- Alarms and notifications
- User management and audit logging

**Out of Scope:**
- Purchase order generation (finance team responsibility)
- Vendor payment processing (handled by finance)
- Fixed asset management (separate system)
- Equipment maintenance scheduling (handled by TMS)

## 5. DETAILED FEATURE SPECIFICATIONS

### Module 1: Item Master Management

#### Feature 1.1: Item Registration

**Description:** Register new DLP items in the system  
**User Role:** Store Keeper, System Administrator

**Workflow:**
1. Click "Add New Item"
2. Enter item details:
   - Part Number (unique identifier)
   - Item Description
   - Category (Traction, Brake, Current Collector, etc.)
   - System Type (which metro train system)
   - Vendor Name (dropdown from master)
   - Unit of Measure (piece, unit, set, etc.)
   - Recommended Stock Quantity
   - Reorder Level
   - Critical Flag (Y/N)
   - Notes/Comments
3. System validates:
   - Part Number uniqueness
   - Mandatory fields completion
   - Numeric field validation
4. Save and generate Item ID
5. Display success message

**Data Fields:**
- Item_ID (auto-generated)
- Part_Number (indexed for search)
- Item_Description
- Category
- System_Type
- Vendor_ID (foreign key)
- Unit_of_Measure
- Recommended_Qty
- Reorder_Level
- Critical_Flag
- Active_Flag
- Created_Date
- Created_By
- Updated_Date
- Updated_By

#### Feature 1.2: Item Search & Filter

**Description:** Find items using multiple search criteria

**Search Criteria:**
- Part Number (exact or partial)
- Item Description (text search)
- Category (dropdown filter)
- System Type (dropdown filter)
- Vendor Name (dropdown filter)
- Stock Status (In Stock, Low Stock, Out of Stock)
- Critical Items Only (checkbox)

**Search Results Display:**
- Item ID, Part Number, Description
- System Type, Category, Vendor
- Current Stock, Recommended Qty, Reorder Level
- Status, Last Updated

**Performance Target:** <1 second for searches of 10,000+ items

#### Feature 1.3: Item Update & Maintenance

**Description:** Update item master data

**Allowed Operations:**
- Update item details (description, category, etc.)
- Change recommended/reorder quantities
- Mark item as critical
- Archive obsolete items
- Bulk update (CSV import)

**Change Tracking:**
- Log all changes with timestamp and user ID
- Maintain change history (audit trail)
- Show before/after values

#### Feature 1.4: Item Expiry Management

**Description:** Track expiry dates for time-sensitive items

**Functionality:**
- Set expiry date at item registration or receipt
- Generate expiry warnings (30, 14, 7, 3 days before)
- Auto-flag expired items (cannot issue)
- Generate expiry reports by date range
- Track expiry-related write-offs

### Module 2: Stock Management

#### Feature 2.1: Stock Level Tracking

**Description:** Real-time tracking of item quantities at each location

**Stock Locations:**
- Central Store
- Temporary Store (at maintenance area)
- In Transit
- On-site at maintenance location

**Stock Record Contains:**
- Item ID
- Location
- Received Quantity
- Consumed Quantity
- Available Quantity
- Reserved Quantity (for pending NCR/Job Cards)
- Last Updated Timestamp
- Last Updated By

**Stock Calculation:**
```
Available_Qty = Received_Qty - Consumed_Qty - Reserved_Qty
```

#### Feature 2.2: Stock Alerts

**Description:** Automated alerts for inventory anomalies

**Alert Types:**

1. **Low Stock Alert**
   - Trigger: Available_Qty ≤ Reorder_Level
   - Notification: Email + SMS to Store Manager
   - Action: Auto-create purchase request (pending approval)

2. **Critical Stock Alert**
   - Trigger: Available_Qty = 0 for critical items
   - Notification: Email + SMS + Dashboard notification
   - Escalation: Store Manager → Operations Manager (if not resolved in 2 hours)

3. **Overstock Alert**
   - Trigger: Available_Qty > (Recommended_Qty × 1.5)
   - Notification: Email to Store Manager
   - Suggestion: Plan utilization or return to vendor

4. **Expiry Alert**
   - Trigger: Expiry Date - Current Date ≤ 30 days
   - Notification: Email to Quality Assurance
   - Action: Plan utilization or write-off

5. **Discrepancy Alert**
   - Trigger: Physical count ≠ System count (during audit)
   - Notification: Email + SMS to Store Manager & Manager
   - Action: Investigate and reconcile

#### Feature 2.3: Stock Receipt

**Description:** Record receipt of parts from vendor

**Receipt Process:**
1. Store Keeper clicks "Receive Parts"
2. Enter/scan Purchase Order number
3. Select vendor from dropdown
4. Add items received:
   - Item ID (search/scan)
   - Quantity Received
   - Quality Check (Pass/Fail)
   - Batch/Serial Number (if applicable)
   - Expiry Date (if applicable)
5. Add receipt notes
6. Verify total cost matches PO
7. Click "Confirm Receipt"
8. System updates stock levels
9. Generate receipt documentation

**Validations:**
- PO must exist and not be already received
- Quantities must match PO
- Quality check must be completed
- All mandatory fields filled

**Post-Receipt Actions:**
- Email notification to relevant teams
- Update vendor performance metrics
- Flag quality issues if marked as "Fail"

#### Feature 2.4: Stock Issue

**Description:** Issue parts from inventory for maintenance

**Issue Methods:**

1. **From NCR**
   - Triggered by NCR system notification
   - Auto-select items based on NCR requirement
   - Manual approval required
   - Issue directly from inventory

2. **From Job Card**
   - Triggered by Job Card creation
   - Maintenance team selects required parts
   - Store Keeper approves and issues
   - Link parts to job card task

3. **Manual Issue**
   - Store Keeper manually creates issue
   - Search and select items
   - Enter quantity to issue
   - Assign to maintenance team member
   - Generate issue note

**Issue Workflow:**
```
Issue Request Created
        ↓
Store Keeper Verification
        ↓
Stock Availability Check
  └─ If Available: Issue parts
  └─ If Not Available: Create backorder / Notify Manager
        ↓
Update Stock Levels (Reserved → Consumed)
        ↓
Generate Issue Note & Barcode
        ↓
Gate Post Notification for outward movement
        ↓
Issue Complete
```

**Tracking Information:**
- Issue ID (unique identifier)
- Items issued (part number, qty, serial if critical)
- Issued To (maintenance team member)
- Issued From (location)
- Purpose (NCR/Job Card/Other)
- Timestamp
- Issued By (store keeper)
- Expected Return Date (if applicable)
- Actual Return Status (Returned/Damaged/Lost)

### Module 3: Transaction Management

#### Feature 3.1: Complete Transaction History

**Description:** Maintain detailed log of all inventory transactions

**Transaction Types:**
- Receipt from Vendor
- Issue to Maintenance
- Return from Maintenance
- Transfer between Locations
- Adjustment (physical count correction)
- Write-off (expired/damaged)
- Scrap (end-of-life)

**Transaction Record:**
```
Transaction_ID
Timestamp
Transaction_Type
Item_ID / Part_Number
Quantity_Before
Quantity_After
From_Location
To_Location
Reference_ID (NCR/Job Card/PO)
Initiated_By (User ID)
Approved_By (if required)
Remarks
Audit_Status
```

**Transaction Rules:**
- All transactions require user authentication
- Transactions are immutable (no editing)
- Adjustments require manager approval
- Write-offs require store manager + operations manager approval

#### Feature 3.2: Physical Audit & Reconciliation

**Description:** Conduct physical inventory audit and reconcile with system

**Audit Workflow:**
1. Store Manager initiates audit
2. Generate audit checklist (all items or selected categories)
3. Store Keeper performs physical count
4. Record counted quantities
5. System compares with system records
6. Flag discrepancies (>5% variance)
7. Investigate and document reasons
8. Reconcile (adjust system to physical count with approval)
9. Generate audit report

**Audit Report Contents:**
- Items audited
- Discrepancies (quantity and %)
- Root cause analysis
- Corrective actions
- Sign-off by store keeper and manager

### Module 4: NCR Integration

#### Feature 4.1: NCR-to-Parts Workflow

**Description:** Automatic parts allocation when NCR is created

**Integration Points:**

1. **NCR Creation Webhook**
   ```
   Receive: {
     "ncr_id": "NCR-2026-001234",
     "fault_description": "Brake valve failure",
     "item_required": "II91919/02105",
     "quantity": 1,
     "urgency": "CRITICAL",
     "maintenance_team": "TEAM-02"
   }
   ```

2. **Stock Check & Reservation**
   - Query stock level for required item
   - If Available: Reserve quantity for NCR
   - If Not Available: Create alert for manager

3. **Parts Issuance**
   - Create issue transaction
   - Reserve in inventory system
   - Generate issue note with NCR reference
   - Notify maintenance team

4. **NCR Update**
   ```
   Send: {
     "ncr_id": "NCR-2026-001234",
     "status": "PART_ISSUED",
     "transaction_id": "TXN-10001",
     "issue_date": "2026-03-19T10:30:00Z"
   }
   ```

#### Feature 4.2: NCR Closure & Parts Return

**Description:** Update inventory when NCR is closed

**Process:**
1. NCR system sends closure notification
2. System checks if parts were used or returned
3. If used: Mark consumed
4. If returned: Update available stock
5. Generate NCR closure report
6. Archive transaction

### Module 5: Job Card Integration

#### Feature 5.1: Job Card Parts Allocation

**Description:** Link parts to specific maintenance job cards

**Integration:**
- Job Card system creates maintenance task with required parts
- DLP system reserves parts upon receiving job card
- Parts issued as needed during maintenance
- Parts returned/accounted at job card closure

**Workflow:**
```
Job Card Created
├─ Train: TS-01
├─ Task: Brake System PM
└─ Required Parts: [BRK001, BRK002, KS1001]
        ↓
DLP System Reserves Parts
├─ BRK001: Reserve 2 units
├─ BRK002: Reserve 1 unit
└─ KS1001: Reserve 1 unit
        ↓
Parts Issued to Maintenance Team
├─ Scanned at Gate Post
├─ Custody transferred to team
└─ Time logged
        ↓
Maintenance Completes
├─ Some parts used
├─ Some parts returned
└─ Damages documented
        ↓
Parts Accounted & Returned
├─ Returned items received at store
├─ Scanned at Gate Post
├─ Custody transferred back to store
└─ Stock updated
        ↓
Job Card Closed
├─ Final inventory adjusted
└─ Consumption report generated
```

### Module 6: Gate Post Integration

#### Feature 6.1: Parts Movement Tracking

**Description:** Track parts movement in and out of store

**Gate Post Integration:**

1. **Outward Gate**
   - Parts issued from store
   - Gate post scans part barcode + NCR/Job Card ID
   - Records exit time and person
   - Updates custody to "IN_TRANSIT"
   - Creates gate post record

2. **Inward Gate**
   - Parts returned to store
   - Gate post scans part barcode + condition
   - Records entry time and person
   - If good: Update custody to "RETURNED"
   - If damaged: Flag for quality review
   - Creates return record

#### Feature 6.2: Custody Tracking

**Description:** Maintain chain of custody for critical items

**Custody States:**
```
IN_STORE → IN_TRANSIT → IN_USE → RETURNED → IN_STORE
                                 ↓
                            DAMAGED (Scrap)
                                 ↓
                            WRITE-OFF
```

**Custody Record:**
- Item ID / Serial Number
- Custody From (person/team)
- Custody To (person/team)
- Timestamp
- Location
- Status
- Signature/Approval

## 7. INTEGRATION ARCHITECTURE

### System Integration Diagram

```
┌────────────────────────────────────┐
│   DLP Inventory Management System   │
│  (Core Application - PostgreSQL)    │
└────────────────────────────────────┘
  ├── NCR System (Webhook)
  │   ├─ New NCR created → Reserve parts
  │   ├─ Issue parts automatically
  │   └─ Update NCR status
  │
  ├── Job Card System (API)
  │   ├─ New Job Card → Reserve parts
  │   ├─ Track parts usage
  │   └─ Close Job Card → Update inventory
  │
  ├── Gate Post System (Barcode scan)
  │   ├─ Outward: Record parts leaving
  │   ├─ Inward: Record parts returning
  │   └─ Track custody transfers
  │
  ├── Financial System (ERP)
  │   ├─ Send consumption for costing
  │   ├─ Budget tracking
  │   └─ Vendor invoicing
  │
  └── Notification System
      ├─ Email gateway
      ├─ SMS gateway
      └─ Dashboard alerts
```

### Integration Details

**NCR System Integration:**
- Endpoint: `POST /integrations/ncr-webhook`
- Authentication: API Key (shared with NCR team)
- Frequency: Real-time (as NCR created)
- Payload: NCR details with required items
- Response: Acknowledge + Parts issuance status

**Job Card System Integration:**
- Endpoint: `GET/POST /api/jobcard/{job_id}`
- Authentication: JWT Token
- Frequency: On demand + polling every 5 minutes
- Payload: Job card details with parts requirements
- Response: Parts allocation status

**Gate Post System Integration:**
- Endpoint: `POST /integrations/gatepost-scan`
- Authentication: Basic Auth
- Frequency: Real-time (as scan happens)
- Payload: Barcode, NCR/Job Card ID, scan type (in/out)
- Response: Acknowledgment + Gate pass
